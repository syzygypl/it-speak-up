Static vs dynamic typing

Statically typed language - A language is statically typed if the type of a variable is known at compile time. (c) Stackoverflow

Dynamically typed language - A language is dynamically typed if the type of value held by a variable is know in run-time.
(c) Guess who

Examples:


  • Static: C, C++, Java
  • Dynamic: Perl, Ruby, Python, Javascript, PHP
  • Untyped: assembly

Usually types system is associated with programming language type


static -> static language

dynamic -> dynamic language (scripting)

Main features of dynamic typing

  • No compilation You don’t have to compile code you can just run it.(In case of static language you can reduce compile time with `incremental compilation`)

  • Can pretend static There are some techniques like `JsDoc` that can increase static analysis purpose and IDE support. 

  • Faster coding
    You can do whatever you want. And write code faster.

  • Easy to understand

Main features of static typing

  • You cannot change variable type once you declare `string` it will be `string` till the end. Assigning value of another type will lead to Exception.

  • Functions have contracts (strictly defined input parameters and output types) which gives us better understanding about what function does. Also it gives us so called `parameter overloading` which may be useful sometimes. Python offers overloading mechanism too because of it strict typing.

  • May also offer `type inference` mechanism of determining variable type in compile time by type of initially assigned value. Which may reduce a bit code producing time and remove unnecessary boilerplate.

But do we really can write code faster?


Usually yes, but sometimes no.


It is easy to understand why we can write code faster in dynamically typed language.


But what can cause faster development in case of statically typed language?

  • Fewer bugs in produced code You know about wrong assignments, function calls, returns immediately after compilation or static analysis. Static analysis can be also applicable to dynamically typed languages but have no such great effect. 

  • You don’t need to know the scope Functions always have strictly declared inputs and output. You don’t have to trace function calls to understand what parameters you got and what you have to do.

  • Writing unit test for static typing takes less time Here we just have to create more test in case of `secure` code. Again with static typing compiler will do the job.

  • Writing secure code is almost impossible or cost a lot of lines If you creating component that have to be secure (sometimes you have to in case of library or business requirements) you have to handle all possible input types and time of producing will dramatically increase. With static typing you code is secure by default (don’t miss with secure business logic).

Statistics Time

  • Some languages have a greater association with defects (bugs) than other languages, although the effect is small. (Closure, Scala are winning one)

  • There is a small but significant relationship between language class and defects. Functional languages are associated with fewer defects than either procedural or scripting languages. (Functional-Static-Strict is the best)

  • There is no general relationship between application domain and language defect proneness.

  • Defect types are strongly associated with languages; some defect type like memory errors and concurrency errors also depend on language primitives. Language matters more for specific categories than it does for defects overall.
