Tests

What’s the main qualities we want to see in our software?

Reliability

It has to be working software, so it can perform it’s main purpose.

Maintainability

It has to have good inner quality, so we can quickly react to changes.

If our software will be reliable

  • our client will have what he expects (and he will be happy)
  • we will not do debugging (and we will be happy)

If our software will be maintainable

  • our client can fulfill his need (and he will be happy)
  • we can write code as at the begging of the project (and we will be happy)

We will not fall into maintenance burden

How we can fulfill this qualities?

  • better tools (frameworks, libraries) that are more reliable than our code.

  • static/strict typing or static analysis to decrease level of programmatic errors.

  • some techniques SOLID, DRY to write better code.

  • some architecture to define layers of responsibility.

  • code review or pair programming to decrease our architectural or logical errors.

But that’s all not really solves the problem.

Tests

  • Quick feedback loop.

  • Test thing that are far away of UI.

  • Describing what production code does.

  • Force you to fix problems in code (when used in deploying lifecycle)

  • Simplifies refactoring.

  • Reduces debugging time.

  • Clarifies code problems.

TDD

Three rules of TDD

  • Don’t write single line of production code without writing failing acceptance test.

  • Don’t write any acceptance test before making production code pass test previous.

  • Don’t write any code additional code after passing acceptance test.

That Keeps You in TDD cycle

Profit

  • More iterative focused approach

  • Clarifies design intentions

  • Always tested code

  • Change monotonic test activity to design
