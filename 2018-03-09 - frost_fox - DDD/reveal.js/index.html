DDD

DDD (Domain Driven Design)

- is an approach or set of techniques of modeling, building and maintaining domain logic intensive applications.

Domain

sphere of knowledge of particular problem that we solve in our application. (Like banking accounting, ship communications, or even programming - depends on application we’re writing)

Model

reflection of the domain in our application. Abstraction layer created by programmers to represent the domain.

Ubiquitous language

a language structured around the domain model and used widely across the team.

The Building blocks of DDD

Entity

  • Represents domain object that has identity

  • The purpose is to represent complex variable with lifecycle and persistance

  • Must be mutable to perform transitions between states

  • Should be always in valid (for domain) state

  • Like User which can be identified by email

Value Object

  • Represents domain object that has no identity

  • The purpose is to be represent value

  • Must be immutable to protect from sharing issue

  • Should be always in valid (for domain) state

  • Like Address which is identifiable

Aggregate

  • Represents cluster of domain objects that can be treated as a single unit

  • The purpose is maintaining invariations and limit complexity

  • Have so called “aggregate root”

General points about objects

  • Avoid traversal reference

  • Use constrained associations

  • Sometimes Entity Sometimes VO ...

Factory

  • Responsible for creating Entities and VO

  • The purpose is maintaining invariations and limit complexity

  • Useful in case of complex aggregates

  • Object with many relations should not be responsible for creating itself (imagine self-created engine)

Repository

  • Responsibility of fetching and storing objects in persistence layer (DB, external API)

  • Repository could be created only for aggregate root or standalone entity

  • Repository can use factories to maintain invariations

  • Almost always has method to get by unique identifier

Service

  • Represents available actions in the domain

  • Can be query or command

  • Like banking transaction service responsible for transferring money between accounts
